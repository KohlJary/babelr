## Getting started

Welcome! This page is our team's collective memory — we keep onboarding notes, conventions, and running jokes here so new folks can get up to speed without pinging everyone in #general. Read through it, and don't hesitate to edit anything that looks stale.

## How we ask for help

If you get stuck on something, don't suffer in silence. Open a thread in #ayuda-general and describe what you tried before asking — that saves everyone time. We don't bite here, and we'd much rather you ask something "silly" than spend three hours searching Google. Nobody was born knowing everything.

## Code conventions

We follow one simple rule: code should be readable before it's clever. Explicit variable names are better than a comment explaining an abbreviation. Code reviews are a dialogue, not an exam — if someone suggests a change, it's to improve the product, never to call you out personally.

## Code Reviews

Reviews should happen quickly — ideally within one business day. Don't hold anyone up longer than necessary. When you leave a comment, always explain the why, not just the what. "That's wrong" doesn't help anyone; "That's wrong because X fails under condition Y" gives the author something to work with.

## About breaks

Don't forget to take breaks. Staring at the screen for hours on end makes your code worse and makes you feel worse too. Go for a walk, grab a coffee, chat with a coworker. On our team, taking breaks isn't slacking off — it's part of the job.
